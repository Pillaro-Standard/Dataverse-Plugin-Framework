# Deployment Tools

This folder contains generated wrapper scripts for deploying Dataverse plugin assemblies and synchronizing plugin registration metadata defined in code.

The wrappers call the Pillaro Dataverse CLI that is packaged inside the `Pillaro.Dataverse.PluginFramework` NuGet package. You do not need to install or build the CLI separately when consuming the package from NuGet.

## Generated files

After installing or updating the package and rebuilding the plugin project, the package creates or updates:

```text
Tools/
  Deployment/
    DeployPlugins.bat
    DeployPlugins.ps1
    README.md

PillaroSettings.json
```

`PillaroSettings.json` is created only when it does not already exist. Existing settings are not overwritten.

## Quick start

From the plugin project root, run:

```bat
.\Tools\Deployment\DeployPlugins.bat
```

The batch wrapper uses the `debug` profile by default.

To use another profile:

```bat
.\Tools\Deployment\DeployPlugins.bat release
```

PowerShell variant:

```powershell
.\Tools\Deployment\DeployPlugins.ps1 -Profile debug
```

Release profile:

```powershell
.\Tools\Deployment\DeployPlugins.ps1 -Profile release
```

The deployment script:

1. Reads `PillaroSettings.json`.
2. Resolves the selected deployment profile.
3. Reads the Dataverse connection string from the configured environment variable.
4. Loads the plugin assembly.
5. Discovers plugin registration metadata from `Register(IPluginRegistration registration)`.
6. Deploys the plugin assembly.
7. Synchronizes Dataverse plugin steps and images.

## CLI help

Normal usage should go through the generated wrappers:

```bat
.\Tools\Deployment\DeployPlugins.bat
```

```powershell
.\Tools\Deployment\DeployPlugins.ps1 -Profile release
```

Configuration is stored in `PillaroSettings.json`. Direct CLI deployment uses:

```text
pillaro-dv deploy [options]
```

The supported direct CLI options are only:

| Option | Description |
|---|---|
| `--settings <path>` | Path to `PillaroSettings.json`. |
| `--profile <name>` | Deployment profile from `PillaroSettings.json`. |
| `--just-assembly` | Deploy only the plugin assembly and skip step/image synchronization. |
| `-h`, `--help` | Show help. |

The solution unique name and plugin assembly path are not CLI arguments.

- solution unique name comes from `solution`
- plugin assembly path comes from `profiles.<profile>.pluginAssemblyPath`
- Dataverse connection string variable comes from `dataverse.connectionStringEnvironmentVariable`

## Configuration

`PillaroSettings.json` lives in the plugin project root.

Example:

```json
{
  "schemaVersion": 1,
  "solution": "MyDataverseSolution",
  "dataverse": {
    "connectionStringEnvironmentVariable": "DV_CONN"
  },
  "defaultProfile": "debug",
  "profiles": {
    "debug": {
      "pluginAssemblyPath": "bin/Debug/MySolution.Plugins.dll"
    },
    "release": {
      "pluginAssemblyPath": "bin/Release/MySolution.Plugins.dll"
    },
    "ci": {
      "pluginAssemblyPath": "${BUILD_BINARIESDIRECTORY}/MySolution.Plugins.dll"
    }
  }
}
```

## Required settings

| Field | Description |
|---|---|
| `solution` | Dataverse solution unique name. Deployed plugin components are added to this solution. |
| `dataverse.connectionStringEnvironmentVariable` | Name of the environment variable containing the Dataverse connection string. Default is `DV_CONN`. |
| `defaultProfile` | Profile used when no profile is passed explicitly. |
| `profiles.<name>.pluginAssemblyPath` | Path to the built plugin DLL for the selected profile. |

Set `pluginAssemblyPath` to the final deployable plugin assembly.

Common examples:

```text
bin/Debug/MySolution.Plugins.dll
bin/Release/MySolution.Plugins.dll
bin/Debug/net462/MySolution.Plugins.dll
bin/Release/net462/MySolution.Plugins.dll
${BUILD_BINARIESDIRECTORY}/MySolution.Plugins.dll
```

Relative paths in `PillaroSettings.json` are resolved from the folder where `PillaroSettings.json` is located.

## Connection string

The deployment tool reads the Dataverse connection string from an environment variable.

Default variable name:

```text
DV_CONN
```

PowerShell, current terminal session:

```powershell
$env:DV_CONN = 'AuthType=ClientSecret;Url=https://your-org.crm4.dynamics.com;ClientId=your-client-id;ClientSecret=your-client-secret;TenantId=your-tenant-id'
```

PowerShell, current Windows user:

```powershell
[Environment]::SetEnvironmentVariable(
    'DV_CONN',
    'AuthType=ClientSecret;Url=https://your-org.crm4.dynamics.com;ClientId=your-client-id;ClientSecret=your-client-secret;TenantId=your-tenant-id',
    'User')
```

Restart Visual Studio, PowerShell or your terminal after setting a user-level environment variable.

Do not commit connection strings or secrets into source control.

## Azure DevOps usage

Store the Dataverse connection string as a secret variable, for example `DataverseConnectionString`, and map it to the environment variable expected by `PillaroSettings.json`.

Example:

```yaml
- powershell: |
    .\MySolution.Plugins\Tools\Deployment\DeployPlugins.ps1 -Profile ci
  displayName: Deploy Dataverse plugins
  env:
    DV_CONN: $(DataverseConnectionString)
```

The `ci` profile should point to the plugin DLL produced by the pipeline build.

Example `ci` profile:

```json
{
  "ci": {
    "pluginAssemblyPath": "${BUILD_BINARIESDIRECTORY}/MySolution.Plugins.dll"
  }
}
```

Environment tokens such as `${BUILD_BINARIESDIRECTORY}` are resolved at deployment time.

## Troubleshooting

| Problem | What to check |
|---|---|
| `Tools/Deployment` is missing | Rebuild the plugin project after installing or updating the NuGet package. |
| `PillaroSettings.json` is missing | Rebuild the plugin project. The file is created only if it does not already exist. |
| CLI not found | Restore NuGet packages and rebuild the plugin project so the wrapper scripts are regenerated. |
| Assembly not found | Check `profiles.<name>.pluginAssemblyPath` for the selected profile. |
| Connection string not found | Check that the environment variable name matches `dataverse.connectionStringEnvironmentVariable`. |
| Visual Studio cannot see the variable | Restart Visual Studio after setting a user-level environment variable. |
| Solution not found | Check that `solution` contains an existing Dataverse solution unique name. |
| Steps or images are not updated | Check that the plugin assembly contains `Register(IPluginRegistration registration)` metadata and that the selected assembly path points to the latest built DLL. |

## Notes

`RegisterTask(...)` in the plugin constructor controls runtime task execution.

`Register(IPluginRegistration registration)` controls deployment metadata for Dataverse steps and images.

Both declarations are intentionally separate and must be kept aligned when changing plugin behavior or deployment registration.
