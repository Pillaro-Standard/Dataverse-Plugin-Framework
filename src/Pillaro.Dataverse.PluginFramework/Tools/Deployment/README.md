# Deployment Tools

This folder contains lightweight wrapper scripts that invoke the Pillaro Dataverse CLI from the NuGet package. Use the PowerShell wrapper in CI; the .bat wrapper is convenient for local Windows runs.

## Quick start
From project root (Windows):

    .\Tools\Deployment\DeployPlugins.bat

Or using PowerShell with a specific profile:

    .\Tools\Deployment\DeployPlugins.ps1 -Profile release

> [!IMPORTANT]
> Do not store connection strings or secrets in committed files.

## Configuration
PillaroSettings.json lives in the project root and controls deployment profiles. It is user-owned and never overwritten by the package.

Example profile structure:

~~~json
{
  "schemaVersion": 1,
  "solution": "MyDataverseSolution",
  "dataverse": {
    "connectionStringEnvironmentVariable": "DV_CONN"
  },
  "defaultProfile": "debug",
  "profiles": {
    "debug": {
      "pluginAssemblyPath": "bin/Debug/MyPlugin.dll"
    },
    "release": {
      "pluginAssemblyPath": "bin/Release/MyPlugin.dll"
    },
    "ci": {
      "pluginAssemblyPath": "${BUILD_BINARIESDIRECTORY}/MyPlugin.dll"
    }
  }
}
~~~

## Local environment
Set the Dataverse connection string in the user environment (PowerShell):

    [Environment]::SetEnvironmentVariable('DV_CONN', 'AuthType=ClientSecret;Url=https://your-org.crm4.dynamics.com;ClientId=your-client-id;ClientSecret=your-client-secret', 'User')

Restart Visual Studio, PowerShell or your terminal after setting the variable.

Temporary session-only setup:

    $env:DV_CONN = 'AuthType=ClientSecret;Url=https://your-org.crm4.dynamics.com;ClientId=your-client-id;ClientSecret=your-client-secret'

Safe verification:

    [Environment]::GetEnvironmentVariable('DV_CONN', 'User') -ne $null

## Azure DevOps
Example pipeline step:

~~~yaml
- powershell: |
    .\Tools\Deployment\DeployPlugins.ps1 -Profile ci
  displayName: Deploy Dataverse plugins
  env:
    DV_CONN: $(DataverseConnectionString)
~~~

**Important:**
- Configure `DataverseConnectionString` as a secret variable or secret variable group
- Map it explicitly into the step's `env` block
- Do not pass connection strings as command-line arguments
- CI profiles can use `${BUILD_BINARIESDIRECTORY}` which is resolved at deploy time

## Troubleshooting

**CLI not found error:**
- Verify the NuGet package is restored correctly
- Check that the wrapper scripts point to the correct CLI path inside the package

**Connection fails:**
- Verify the environment variable is set correctly
- Check the connection string format
- Ensure the service principal or user has appropriate permissions

**Deployment artifacts missing:**
- Ensure the plugin assembly is built and merged before deployment
- Check the `pluginAssemblyPath` in PillaroSettings.json points to the correct output
