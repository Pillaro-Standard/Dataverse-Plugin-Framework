# Plugin Deployment Flow

This document describes the supported end-to-end plugin deployment flow.

The deployment flow is designed to be simple: build the plugin project, keep a small settings file next to it, and run one command from the plugin project folder.

Plugin deployment is implemented by Pillaro CLI through the Dataverse SDK. It does not use PAC.

## Supported Command

Run from the plugin project folder:

```bat
pillaro-dv plugin deploy --settings PillaroSettings.json --confirm
```

The command:

1. reads `PillaroSettings.json`,
2. performs preflight checks,
3. discovers registration metadata from `PluginBase` plugin classes,
4. validates the generated manifest,
5. connects to Dataverse using the configured environment variable,
6. creates or updates the plugin assembly,
7. creates missing plugin type records,
8. reads current Dataverse registration state for those plugin types,
9. calculates the registration diff,
10. creates, updates or deletes steps and images,
11. ensures supported solution membership.

## PillaroSettings.json

Minimum supported shape:

```json
{
  "solution": "PillaroPluginFrameworkExamples",
  "dataverse": {
    "connectionStringEnvironmentVariable": "DV_CONN"
  },
  "plugins": {
    "assembly": "bin/Debug/Pillaro.Dataverse.PluginFramework.Examples.Plugins.dll"
  }
}
```

Rules:

- `solution` is required for plugin deployment.
- `plugins.assembly` is required.
- `dataverse.connectionStringEnvironmentVariable` defaults to `DV_CONN` when omitted.
- Relative paths are resolved from the current working directory.
- The settings file does not contain a plugin assembly ID.
- The settings file does not contain a PAC profile for plugin deployment.
- The settings file does not contain a manifest path for plugin deployment.

The manifest is an internal artifact generated by the deploy command and written to:

```text
artifacts/plugin-manifest.json
```

## Dataverse Connection

Set the configured environment variable to a Dataverse SDK connection string before running deployment:

```bat
set "DV_CONN=AuthType=ClientSecret;Url=https://org.crm4.dynamics.com;ClientId=...;ClientSecret=...;TenantId=..."
```

The variable name can be changed in `PillaroSettings.json`:

```json
{
  "dataverse": {
    "connectionStringEnvironmentVariable": "CONTOSO_DV_CONN"
  }
}
```

## Preflight

Before any Dataverse write, the command prints a checklist:

```text

Settings: C:\src\Contoso.Plugins\PillaroSettings.json

Configuration:
  Solution: ContosoCore
  Plugin assembly: bin/Debug/Contoso.Plugins.dll
  Resolved plugin assembly: C:\src\Contoso.Plugins\bin\Debug\Contoso.Plugins.dll
  Connection string env: DV_CONN

  [OK] settings file found
  [OK] solution configured
  [OK] plugin assembly configured
  [OK] plugin assembly found
  [OK] Dataverse connection string found in DV_CONN
```

If any prerequisite fails, deployment stops before connecting to Dataverse.

## Source of Truth

The source of truth is the plugin class `Register(IPluginRegistration registration)` method:

```csharp
public sealed class ContactPlugin : PluginBase
{
    public ContactPlugin(string unsecureConfig, string secureConfig)
        : base(unsecureConfig, secureConfig)
    {
    }

    public override void Register(IPluginRegistration registration)
    {
        registration
            .OnUpdate<Contact>("5072086e-1508-f111-8407-000d3ab261ac")
            .PreOperation()
            .Synchronous()
            .Rank(1)
            .WhenChanged(c => c.FirstName, c => c.LastName)
            .WithPreImage(
                "7f8a44bb-4d4f-4cd9-9e22-efb1472a1001",
                "PreImage",
                c => c.FirstName,
                c => c.LastName);
    }
}
```

The step ID is expected to match `sdkmessageprocessingstepid`.

The image ID is expected to match `sdkmessageprocessingstepimageid`.

## SDK Deployment Responsibilities

Pillaro CLI deploys these Dataverse records directly through the SDK:

- `pluginassembly`
- `plugintype`
- `sdkmessageprocessingstep`
- `sdkmessageprocessingstepimage`

For the plugin assembly, the deployer:

- reads the DLL bytes,
- resolves the .NET assembly name from the DLL,
- finds an existing `pluginassembly` by name,
- creates it when missing,
- updates the assembly content when present.

For plugin types, the deployer:

- uses the manifest plugin type names,
- creates missing `plugintype` records for the deployed assembly,
- reuses existing `plugintype` records when present.

For steps and images, the deployer reuses the Pillaro registration manifest and diff logic:

- reads current Dataverse registration state for deployed plugin types,
- calculates create/update/delete/ok changes,
- applies required step and image metadata changes.

## Solution Membership

The deploy command ensures solution membership for:

- `pluginassembly`
- `sdkmessageprocessingstep`

Plugin type records are deployed as assembly metadata. Image records are deployed as step metadata.

## PAC Boundary

Plugin deployment does not use PAC. PAC can still be used by other tooling, such as early-bound generation or solution operations outside this deploy command.

## Safe Defaults

- Preflight runs before any Dataverse write.
- Manifest validation must pass before connection/deployment.
- `--confirm` is required when registered metadata marks a step as requiring confirmation.
- Deployment scope is limited to plugin types discovered from the current assembly.
- Relative paths are resolved from the current working directory.
- `earlyBound.solution` is not used as a fallback for plugin deployment.

## Local Example

From the plugin project folder:

```bat
set "DV_CONN=AuthType=ClientSecret;Url=https://org.crm4.dynamics.com;ClientId=...;ClientSecret=...;TenantId=..."
dotnet build
C:\tools\pillaro-dv.exe plugin deploy --settings PillaroSettings.json --confirm
```

## Pipeline Example

```yaml
steps:
- script: dotnet build src/Contoso.Plugins/Contoso.Plugins.csproj -c Release
  displayName: Build plugins

- script: pillaro-dv plugin deploy --settings src/Contoso.Plugins/PillaroSettings.json --confirm
  displayName: Deploy Dataverse plugins
  env:
    DV_CONN: $(DataverseConnectionString)
```
